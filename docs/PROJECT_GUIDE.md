# FixariVan — описание проекта, инструкция и рекомендации

## 1. Что это за проект

**FixariVan** — веб-приложение для мобильного сервиса по ремонту техники (контекст: Финляндия, реквизиты и языки ru/en/fi). Оно объединяет в одном месте:

- **Документы**: акт приёма, квитанция об оплате, счёт (invoice), диагностические отчёты (мобильные / ПК).
- **Клиенты**: справочник в SQLite, поиск, привязка к заказам и документам.
- **Операции**: склад и движения, календарь событий, трекинг документов по клиентам, **финансовый обзор** (доход/расход/НДС и др.).
- **Заметки** мастера (отдельный раздел, стиль «Заметки» iOS).
- **Публичные ссылки для клиента**: просмотр акта/квитанции/отчёта по **токену** в URL (без входа в админку).
- **PDF**: генерация через Dompdf (админ-сессия или токен документа).

**Источник данных в продакшене:** SQLite-файл `storage/fixarivan.sqlite`. Резервные JSON в `storage/` поддерживаются политикой данных, но основной режим — SQLite. Наследие MySQL в `config.php` используется только старыми/редкими сценариями.

## 2. Технологии

| Слой | Технологии |
|------|------------|
| Сервер | PHP 8.x, PDO SQLite, сессии PHP |
| Клиент | HTML, CSS, JavaScript (fetch, без обязательного фреймворка) |
| PDF | Dompdf (`api/dompdf/`) |
| Данные | SQLite (WAL, foreign keys, busy_timeout в коде) |

Конфигурация секретов: **`config.local.php`** (копия с `config.example.php`), не коммитить в открытый репозиторий.

## 3. Архитектура доступа (кратко)

Подробно: **[ACCESS_MODEL.md](ACCESS_MODEL.md)**.

| Режим | Как работает |
|-------|----------------|
| **Админ** | После `admin/login.php` — cookie сессии `admin_logged_in`. Страницы с `FIXARIVAN_REQUIRE_ADMIN_SESSION` + `auth_check.js` проверяют сессию через API. |
| **Клиент (viewer)** | Страницы `order_view.php`, `receipt_view.php`, `invoice_view.php`, `report_view.php` — только по **валидному токену** в ссылке. |
| **PDF** | Либо админ-сессия, либо в запросе `clientToken`, совпадающий с записью в SQLite для этого документа. |

**Важно:** флаги в `localStorage` не являются авторизацией; чувствительные операции только на сервере.

## 4. Инструкция: с нуля до работы

### 4.1. Установка

1. Скопировать **`config.example.php` → `config.local.php`**, заполнить как минимум `ADMIN_USERNAME` / `ADMIN_PASSWORD` (вход в админку).
2. Убедиться, что каталог **`storage/`** существует и **доступен для записи** веб-серверу (создаётся `fixarivan.sqlite` при первом обращении к API с SQLite).
3. Расширения PHP: **`pdo_sqlite`**, желательно **`mbstring`**, для PDF — **`dom`**, при необходимости **`gd`** / **`zip`** (см. `api/health.php`).
4. Открыть в браузере **`admin/login.php`**, войти, затем **`index.php`** (рабочий стол / `dashboard_app.html`).

### 4.2. Ежедневная работа

- **Рабочий стол** (`index.php`): быстрые переходы к актам, квитанциям, счетам, диагностике, клиентам, финансам, треку, календарю, складу, заметкам.
- **Квитанция** можно оформить **без акта** — поле заказа необязательно; обязательны клиент, суммы/оплата по логике формы и подпись мастера.
- **Клиенты**: заводите карточки в разделе клиентов; для старых контактов (например с WhatsApp) удобно нормализовать телефон в одном формате.
- **Финансы**: период, KPI, детализации и экспорт — по интерфейсу `finance.html`.
- **Заметки**: `notes.html`, данные в таблице `notes` в SQLite.

### 4.3. Проверка работоспособности

- **`api/health.json`** не существует — использовать **`api/health.php`**: JSON о расширениях, записи в `storage/`, доступности SQLite.
- С подсчётом документов (акты/квитанции/отчёты) — **`api/db_status.php`** (нужна **админ-сессия**).

### 4.4. Резервное копирование

- Критично: каталог **`storage/`** целиком (включая **`fixarivan.sqlite`** и при необходимости `admin_auth.json`, профили компании).
- В репозитории есть CLI **`tools/backup.php`** — можно вызывать по cron с выгрузкой архива **вне сервера** (облако, локальный ПК).

### 4.5. Деплой (например cPanel)

Сводный чеклист: **[STATUS.md](STATUS.md)**. Кратко: HTTPS, права на `storage/`, PHP-лимиты через панель/`.user.ini`, не заливать диагностические скрипты в прод.

## 5. Структура репозитория (логически)

| Путь | Назначение |
|------|------------|
| `index.php`, `dashboard_app.html` | Рабочий стол (после входа) |
| `admin/` | Вход, настройки, logout |
| `api/` | JSON API, сохранение документов, финансы, клиенты, заметки, PDF и т.д. |
| `api/sqlite.php`, `api/lib/` | Подключение БД, шаблоны документов, безопасность PDF, финансовая логика |
| `storage/` | SQLite, токены, временные файлы — **не отдавать напрямую** (есть `.htaccess`) |
| `docs/` | Модель доступа, статус, это руководство |
| `css/` | Общие темы оформления (например `fixarivan-theme.css`) |

Каталог эндпоинтов: **`api/index.php`** (GET без секретов).

## 6. Рекомендации

### Безопасность

- Сильный пароль админа; после смены в **настройках** предпочтительно хранение в **`storage/admin_auth.json`** (bcrypt), а не только константы в `config.local.php`.
- Только **HTTPS** в проде; ограничить брутфорс входа (лимиты хостинга, по возможности задержка/капча на `/admin/`).
- Не публиковать **`config.local.php`**, не коммитить его в git.
- Публичные ссылки с токенами воспринимать как **секретные** — не светить в открытых каналах без необходимости.

### Эксплуатация

- Регулярные **бэкапы** `storage/`.
- Следить за **местом на диске** и логами хостинга.
- После обновления PHP — проверить **`api/health.php`**.

### Разработка

- Новые фичи с записью в БД — через **`ensureSqliteSchema`** в `api/sqlite.php` и миграции по образцу существующих таблиц.
- API для админки — подключать **`require_admin_session.php`**, не полагаться на JS-«флаги» авторизации.

### Ограничения

- **SQLite** на shared-хостинге: очень высокая одновременная запись с многих воркеров может давать редкие блокировки; для одной мастерской нагрузки обычно достаточно.
- Старые страницы **`admin_*.html`** — вспомогательные; основной сценарий — дашборд и современные разделы.
