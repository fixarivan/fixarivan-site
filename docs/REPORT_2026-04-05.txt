1. Краткое резюме
--------------------------------------------------------------------------------

  • Единый центр заказа: SQLite + JSON-бэкап, client_token, связь с клиентами,
    складом (order_lines → order_warehouse_lines), календарём закупок (supply),
    порталом.

  • Потоки order_new ↔ save_order_fixed ↔ Track согласованы: заявка на закуп
    (supply_request) и синхронизация склада строятся из структурированных
    позиций order_lines_json; текстовый блок «Заявка на склад» в Track снят.

  • Склад ↔ заказ: supply_request и supply_urgency хранятся в таблице orders;
    строки со склада учитываются по явному from_stock, а не только по
    inventory_item_id; напоминания AUTO_SUPPLY в календаре требуют
    public_expected_date.

  • Track: при вводе артикула (SKU) — подсказки из каталога склада, автоподстановка
    наименования и цен (закуп/продажа) при точном совпадении SKU или выборе из
    списка; при сохранении и при GET заказа строки обогащаются inventory_item_id
    по SKU (fixarivan_enrich_order_lines_inventory_ids).

  • Список покупок (inventory.html): из очереди закупок — кнопки «Приход»
    (движение с order_warehouse_line_id) и «Убрать из списка» (скрытие без
    удаления заказа); привязка к карточке склада подтягивается из order_lines_json
    по SKU/имени, если в order_warehouse_lines не было inventory_item_id.

  • Приход по строке заказа: строка переводится в «arrived» только когда
    накопленное количество прихода ≥ qty по строке (order_warehouse_lines.qty_received);
    после полного прихода пересчитывается parts_status / статус заказа
    (fixarivan_recompute_order_parts_aggregate).

  • Агрегат parts_status = partial: публичный статус заказа остаётся waiting_parts
    (не переводится в «в работе»), пока не готовы все строки; чип «частично пришло»
    на портале и в Track согласованы.

  • inventory.html: поле количества в модалке движения — id adjQtyInput (не
    adjustQuantity), чтобы не конфликтовать с функцией submit; при «Приход» из
    списка покупок карточка подгружается по id (ensureInventoryItemForId), если
    её нет в текущем списке из-за поиска/фильтра.

  • Публичные viewer-страницы и портал: ru / en / fi через document_templates
    и viewer_i18n.

  • Финансы: finance.html опирается на api/finance.php и finance_lib.php
    (касса по периоду, расход по parts_purchase_total заказов с датой приёма
    в периоде, сводные KPI).

  • Отчёты диагностики: основные формы — diagnostic_mobile.html и
    diagnostic_pc.html + save_report_fixed.php. Отдельная unified-форма
    diagnostic_report.html удалена как лишняя (нет ссылок, файл не входит в SW).

  • В заказ добавлено поле public_estimated_cost (текст «ориентировочная
    стоимость работы» для портала и шаблона акта).

  • (дополнение 27.03.2026) Календарь AUTO_SUPPLY: при пустой нормализованной
    public_expected_date удаляются «зависшие» события (fixarivan_create_supply_reminder).

  • (дополнение 27.03.2026) Единое сохранение JSON заказа на диск: api/lib/order_json_storage.php
    — save_order_fixed, order_track_order_lines, save_order_supply; пути в order_supply /
    generate_beautiful_client_act согласованы.

  • (дополнение 27.03.2026) P2 блок 7: supply_warning в order_track_patch; Track и order_new
    показывают предупреждение, если есть закупка без public_expected_date.

  • (дополнение 27.03.2026) Портал: порядок «документы → позиции»; строки и суммы через
    order_client_portal.php; единая загрузка featured-заказа с get_document:
    api/lib/order_document_load.php (merge JSON + enrich строк). Исправлен вызов
    fixarivan_portal_public_order_lines в client_portal.php (аудит).

5. Изменения по темам (сводка за период, вкл. 05.04.2026)
--------------------------------------------------------------------------------

5.1 Единый поток заказа (order_new ↔ Track ↔ supply ↔ склад)

  Проблема (устранена): позиции order_lines_json попадали в склад, но цепочка
  «заявка на закупку» (инвентарь + календарь) срабатывала не всегда; get_document
  не отдавал supply-поля из JSON; supply_request не всегда попадал в SQLite при
  сохранении из мастер-формы.

  Решение (базовое):
    • api/lib/order_supply.php — fixarivan_supply_request_from_order_lines(),
      fixarivan_apply_supply_effects(); строка считается «со склада» только при
      явном from_stock (не только при наличии inventory_item_id).
    • api/save_order_fixed.php — запись supply_request / supply_urgency в INSERT
      и ON CONFLICT UPDATE; при сохранении из мастер-формы с позициями заявка
      на закуп пересчитывается из order_lines_json; mergeOrderRecord подмешивает
      supply_* из overlay.
    • api/save_order_supply.php — тот же fixarivan_apply_supply_effects().
    • api/get_document.php — fixarivan_merge_order_json_overlay() для supply_*.
    • api/order_track_order_lines.php — обновление orders.supply_request из
      позиций при сохранении из Track.

  Напоминание: событие календаря AUTO_SUPPLY создаётся при наличии
  public_expected_date у заказа; без даты напоминание не ставится.

5.2 Track (track.html)

  Поиск, фильтры, статус для клиента (SELECT + order_track_patch), внутренний
  комментарий, быстрые ссылки на квитанцию/счёт/отчёты (mobile/pc), портал,
  подсветка просрочки по public_expected_date при waiting_parts.

  Таблица «Позиции заказа»: сохранение через order_track_order_lines.php;
  списание со склада по галочке «Со склада» при совпадении SKU или
  inventory_item_id.

  Удалён устаревший UI «Заявка на склад» (ручной текст name | qty | type |
  note и кнопка «Сохранить заявку»): единый источник — позиции заказа; подсказка
  под таблицей про синхронизацию склада и необходимость даты для календаря.

  Подсказки по артикулу: при вводе SKU запрос к api/inventory_list.php?q=…
  (выпадающий список: наименование, остаток); под полем — строка «На складе:
  N шт.»; при выборе из списка или при точном совпадении SKU после blur
  подставляются наименование, закуп € и продажа € со склада; inventory_item_id
  в data-inv-id и в сохраняемом JSON. Перед «Сохранить позиции» для всех строк
  вызывается refreshSkuHintForRow — чтобы не терять привязку без blur.
  Для строк уже списанных со склада поле SKU только для чтения.
  get_document при загрузке заказа обогащает order_lines_json id по SKU (как при save).

  Обновление (ранее по проекту): ссылка «Отчёт (форма)» (diagnostic_report.html)
  удалена — остаются только отчёты тел/планшет и ПК.

5.3 order_new.html

  Подсказки по позициям и ремонту; после сохранения — ссылка клиенту на портал

  стоимость работы (publicEstimatedCost → public_estimated_cost).

  Prefill из query: client_name, client_phone, client_email.

5.4 Схема orders (SQLite)

  добавляются supply_request TEXT и supply_urgency TEXT (согласованность с
  закупкой и календарём при хранении только в SQLite).

  save_order_fixed.php, order_center.php — запись поля в INSERT orders.

5.11 Дополнение 27.03.2026 (итерация 2) — модалка прихода, статусы, поиск на складе

  Проблемы (устранены)
    • Кнопка «Применить» в модалке прихода не вызывала API: у input был id
      adjustQuantity — в window перекрывалась глобальная функция adjustQuantity
      из inventory_sqlite_override.js. Решение: id поля → adjQtyInput; submit
      вызывает window.adjustQuantity().
    • «Приход» из списка покупок при активном узком поиске: позиция с нужным
      inventory_item_id отсутствовала в массиве inventory (loadInventory отдаёт
      только совпадения по q). Решение: ensureInventoryItemForId() — GET
      inventory_list.php?id=…, merge в inventory; вызывается в
      openPurchaseArrivalFromQueue и в adjustQuantity перед движением.
    • Частичная комплектация (одна строка со склада, другая в закупке) переводила
      заказ в «в работе». Решение: в fixarivan_recompute_order_parts_aggregate
      при agg === partial → order_status / public_status = waiting_parts (как при
      waiting); in_progress при полной готовности строк (ready).
    • В Track не отображался агрегат запчастей (на портале — «частично пришло»).
      Решение: в groupDocuments передаётся parts_status из orders; в карточке заказа
      бейдж «Запчасти: …» с теми же смыслами, что чип на client_portal.php.
