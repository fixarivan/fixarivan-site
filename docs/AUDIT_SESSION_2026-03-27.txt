         generate_beautiful_client_act.php — чтение каталога заказов

  Суть: supply_warning в ответе order_track_patch; алерты в Track при смене статуса,
         языка, даты; order_new показывает supply_warning после save_order_fixed
  Файлы: api/order_track_patch.php, track.html, order_new.html

Тема: Единая загрузка заказа

  Файлы: api/lib/order_document_load.php (новый); get_document.php; client_portal.php


Раздел 3. ЛОГИЧЕСКИЕ ЦЕПОЧКИ (ПРОВЕРКА)
--------------------------------------------------------------------------------

3.1 Supply → календарь
  • fixarivan_apply_supply_effects → fixarivan_create_supply_reminder
  • Пустая дата: DELETE AUTO_SUPPLY и return — согласовано с order_track_patch
    (fixarivan_clear_supply_calendar_for_order при очистке даты).
  • Остаточный риск: если supply_request непустой, но парсер дал пустой список
    позиций, apply_supply_effects выходит раньше create_supply_reminder — теоретически
    возможны «зависшие» события при исключительно битом тексте заявки (узкий кейс).

3.2 Сохранение заказа → JSON
  • Все основные пути используют fixarivan_save_order_json_files и единые каталоги.

3.3 order_track_patch
  • UPDATE заказа → при изменении только ожидаемой даты: clear календаря или
    fixarivan_apply_supply_effects.
  • supply_warning: финальный expVal + supply_request и order_lines_json из первой
    выборки (PATCH эти поля не меняет) — корректно для предупреждения о закупке без даты.

3.4 get_document ↔ портал
  • Оба используют fixarivan_load_order_from_sqlite_or_json.
  • Fallback без строки в SQLite: только JSON-файл заказа (без enrich, как и раньше
    по смыслу fallback в API).

Раздел 5. ЗАМЕЧАНИЯ И ТЕХДОЛГ
--------------------------------------------------------------------------------

  • get_document.php: путь к JSON в loadJsonDocument для заказа дублирует логику
    fixarivan_orders_storage_dir() — косметика, на поведение не влияет.
  • client_portal.php: подключение order_document_load тянет order_warehouse_sync
    и далее тяжёлые зависимости — больше памяти/времени на запрос; для одного мастера
    обычно приемлемо.
  • docs/REPORT_2026-04-05.txt упоминает fixarivan_merge_order_json_overlay в
    get_document.php — после рефакторинга функция перенесена в order_document_load.php
    (при необходимости обновить документ вручную).
  • Автотесты в репозитории (tests/…) в рамках этого аудита на машине разработчика
    не запускались — рекомендуется прогон после деплоя.


Раздел 6. ОЦЕНКА ГОТОВНОСТИ
--------------------------------------------------------------------------------

Критерий                              | Оценка
--------------------------------------|------------------------------------------
Согласованность supply / дата /       | Хорошо; редкий кейс «непустой текст заявки,
  календарь                            | 0 строк после парсера» не закрыт полностью
Единый слой JSON на диск              | Хорошо
Предупреждения (Track / order_new /    | Хорошо
  patch)
Портал: структура и единый loader      | Хорошо после исправления имени функции
Регрессии                             | Одна критическая ошибка портала устранена

Итог:
  • Для пилотной эксплуатации одним мастером — готовность высокая при условии
    ручной проверки: портал по token, переключение заказа, документы и таблица
    позиций; Track — смена даты/статуса и supply_warning при закупке без даты;
    сохранение заказа из мастер-формы и Track.
  • Для продакшена со SLA — желательны smoke-тесты (curl/Playwright) на портал и
    get_document; по желанию — интеграционный тест сценария «очистка даты + заявка
    на закуп».
