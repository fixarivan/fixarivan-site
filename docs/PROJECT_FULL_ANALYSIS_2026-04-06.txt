1. КРАТКИЙ ВЫВОД
--------------------------------------------------------------------------------

FixariVan уже выглядит как рабочая единая система для небольшой сервисной
мастерской: заказы, склад, клиенты, документы, публичные viewer-ссылки, PDF,
финансы, заметки и календарь собраны в одном проекте на PHP + SQLite.

Сильная сторона проекта: практичная вертикальная интеграция.
Слабая сторона проекта: высокая связанность логики между order/inventory/document
flow, наличие legacy-слоёв и большой объём бизнес-логики в отдельных PHP-файлах
без единой сервисной границы.

На текущем этапе проект уже не выглядит как набор разрозненных форм. Это скорее
MVP/production-like monolith для одного сервиса. Основной риск не в отсутствии
функций, а в консистентности сложных сценариев, где пересекаются:

  заказ -> order_lines_json -> order_warehouse_lines -> inventory_movements
  -> Track -> receipt/invoice/report -> client portal -> finance

--------------------------------------------------------------------------------
2. ОБЩАЯ АРХИТЕКТУРА
--------------------------------------------------------------------------------

Технологический стек:
  - PHP 8.x
  - PDO SQLite
  - HTML / CSS / JavaScript без тяжёлого фронтенд-фреймворка
  - Dompdf для PDF
  - PHP session для admin-доступа
  - token-based viewer links для клиента

Архитектурный тип:
  - server-rendered / multi-page monolith
  - API-слой реализован набором отдельных PHP-endpoint'ов
  - фронтенд состоит из независимых HTML-страниц, связанных через fetch

Основная точка данных:
  - storage/fixarivan.sqlite

Дополнительное хранение:
  - storage/*.json
  - storage/*_tokens/
  - generated_pdfs/
  - generated_pdfs/debug_html/

Вывод:
  - главный source of truth в практике уже SQLite
  - JSON остаётся как legacy/fallback/backup-compatible слой

3. КЛЮЧЕВЫЕ ПОДСИСТЕМЫ
--------------------------------------------------------------------------------

3.1. Заказы

Основные файлы:
  - order_new.html
  - track.html
  - api/save_order_fixed.php
  - api/get_document.php
  - api/lib/order_center.php
  - api/lib/order_warehouse_sync.php
  - api/order_track_order_lines.php

Роль:
  - создание и обновление заказа
  - связь клиента, устройства, статусов, работ, позиций и публичного токена
  - синхронизация заказа со складом и документами

Оценка:
  - это центральная ось всего проекта
  - модуль развитый, но чувствительный к регрессиям
  - любое изменение order flow почти всегда затрагивает 3-5 подсистем

3.2. Склад и закупка

Основные файлы:
  - inventory.html
  - api/inventory_movement.php
  - api/order_purchase_queue.php
  - api/order_purchase_queue_action.php
  - api/lib/inventory_arrival_auto_out.php
  - api/lib/order_warehouse_sync.php
  - js/inventory_sqlite_override.js

Роль:
  - остатки
  - движения IN / OUT
  - очередь "Под заказ"
  - auto-out по заказным позициям
  - связь с Track и заказом

Оценка:
  - один из самых сложных и хрупких блоков проекта
  - здесь уже виден реальный production domain, а не просто каталог товаров
  - логика корректна только при строгой дисциплине связей по order_line_key /
    inventory_item_id и без fallback-угадывания

3.3. Клиенты

Основные файлы:
  - clients.html
  - api/clients.php
  - client_portal.php
  - api/lib/order_client_portal.php

Роль:
  - клиентская карточка
  - заказы клиента
  - действия по документам
  - публичный portal-view

Оценка:
  - UX стал заметно лучше и ближе к CRM-lite
  - архитектурно это уже не просто справочник, а внешний интерфейс заказа

3.4. Документы и PDF

Основные файлы:
  - receipt.html / invoice.html
  - api/save_receipt_ultimate.php
  - api/save_invoice.php
  - api/save_report_fixed.php
  - api/generate_dompdf_fixed.php
  - api/lib/document_templates.php
  - receipt_view.php / invoice_view.php / report_view.php / order_view.php

Роль:
  - создание документов
  - HTML->PDF pipeline
  - web-view по токену
  - локализация viewer/UI

Оценка:
  - одна из самых зрелых подсистем
  - хороший уровень практической завершённости
  - есть уже отдельный слой шаблонов и переводов

3.5. Финансы

Основные файлы:
  - finance.html
  - api/finance.php
  - api/lib/finance_lib.php

Роль:
  - сводка доходов, расходов, invoices, receipts, KPI

Оценка:
  - полезный operational dashboard
  - логика ещё не бухгалтерская, а управленческая
  - именно здесь важна чёткая формализация basis: cash vs order-period

3.6. Дашборд / рабочий стол

Основные файлы:
  - index.php
  - dashboard_app.html

Роль:
  - entry point
  - KPI
  - плитки переходов
  - быстрый доступ к рабочим разделам

Оценка:
  - UI стал существенно чище
  - дашборд теперь ближе к роли "операционного хаба"

3.7. Диагностика

Основные файлы:
  - diagnostic_mobile.html
  - diagnostic_pc.html
  - api/save_report_fixed.php
  - report_view.php

Роль:
  - диагностические формы
  - хранение отчётов
  - выдача клиенту web-view

Оценка:
  - практичный и нужный блок
  - временное отключение PDF для report через UI выглядит разумной защитой от
    лишней сложности, пока основной viewer-сценарий стабилен

4. ЧТО В ПРОЕКТЕ СДЕЛАНО ХОРОШО
--------------------------------------------------------------------------------

1. Есть реальная сквозная бизнес-модель.
   Проект не ограничивается только документами или только складом. Заказ связан
   с клиентом, документами, публичным порталом, складом и аналитикой.

2. SQLite выбран уместно.
   Для одной мастерской это даёт низкий порог поддержки и быстрый деплой.

3. Разделение ADMIN и TOKEN-only в целом понятное.
   Админка на сессии, клиентские viewer links по токену, PDF защищён отдельной
   проверкой.

4. Документный слой уже оформлен как система.
   Есть viewer pages, шаблоны, i18n, генерация PDF, токены и сохранение.

5. Проект хорошо приспособлен под реальную операционную работу.
   Track, inventory, clients, dashboard и portal закрывают ежедневный цикл.

6. Кодовая база уже содержит внутреннюю документацию.
   В docs/ есть guide, status, access-model, прошлые аудиты и тематические отчёты.

5. ОСНОВНЫЕ РИСКИ И ТЕХНИЧЕСКИЙ ДОЛГ
--------------------------------------------------------------------------------

1. Высокая связанность order flow.
   Один и тот же сценарий проходит через order_new, save_order_fixed, Track,
   warehouse sync, purchase queue, inventory movements, receipt autofill,
   portal and finance. Это делает систему мощной, но хрупкой.

2. Legacy/fallback слой ещё не до конца вычищен.
   В проекте всё ещё встречается логика совместимости с JSON overlay, старыми
   viewer/path-паттернами и историческими структурами данных.

3. Риск дрейфа авторизации.
   Большое количество отдельных PHP endpoint'ов повышает шанс, что один файл
   может оказаться слабее защищён, чем остальные.

4. Сложность "склад <-> заказ".
   Это самый чувствительный участок. Любая неточная связь по SKU/имени/notes
   быстро создаёт фантомные очереди, неверный arrival и регрессии статусов.

5. Debug и служебные артефакты.
   generated_pdfs/debug_html и служебные логи полезны для диагностики, но при
   слабом контроле доступа могут стать источником утечки данных.

6. Нет единого application service layer.
   Бизнес-правила распределены между endpoint-файлами и lib-функциями.
   Это ускоряет MVP-развитие, но усложняет долгосрочную поддержку.

7. Финансовая логика требует строгой формализации.
   Даже небольшая разница в понятиях "оплачено", "начислено", "по дате оплаты",
   "по дате приёма заказа" быстро искажает цифры.

6. ТЕКУЩЕЕ СОСТОЯНИЕ ПО ЗРЕЛОСТИ МОДУЛЕЙ
--------------------------------------------------------------------------------

Оценка дана не как "идеально / плохо", а как practical readiness.

  Дашборд / навигация              -> высокий
  Документы / viewer / PDF         -> высокий
  Клиенты / portal                 -> высокий
  Заказы                           -> высокий, но чувствительный
  Склад / order queue / auto-out   -> средне-высокий, но самый рискованный
  Финансы                          -> средний
  Диагностика                      -> средне-высокий
  Безопасность / эксплуатация      -> средне-высокий

Общее впечатление:
  проект уже пригоден для реальной работы, но устойчивость зависит от того,
  насколько аккуратно поддерживаются сложные цепочки order/inventory/finance.

7. ВАЖНЫЕ НАБЛЮДЕНИЯ ПО ДАННЫМ И ХРАНЕНИЮ
--------------------------------------------------------------------------------

1. SQLite — фактический центр системы.
   Это хорошо для простоты сопровождения.

2. JSON-хранилище остаётся полезным, но увеличивает когнитивную нагрузку.
   Нужно всегда понимать: что primary source, а что backup/overlay/compat.

3. Token model жизнеспособна.
   Но токены нужно воспринимать как секретные URL, а не как "просто ссылки".

4. Служебные и generated-данные должны оставаться под контролем.
   Особенно: storage/, generated_pdfs/, logs/.

--------------------------------------------------------------------------------
8. ФИНАНСЫ: ОТДЕЛЬНЫЕ ВЫВОДЫ
--------------------------------------------------------------------------------

На дату отчёта финансовый модуль полезен, но не должен восприниматься как
строгая бухгалтерия.

Что уже улучшено:
  - исправлен разбор старых строковых оценок работ
  - исправлен выбор статуса заказа для completed/in_progress counters
  - summary/tax approx_profit приведён к более согласованной логике по cash flow

Что остаётся зоной внимания:
  - возможный double count, если один и тот же денежный поток отражён и через
    paid receipt, и через paid invoice
  - необходимость явно договориться, что именно значит revenue/expense в UI:
    cash, accrual, order estimate или hybrid KPI

Вывод:
  finance module полезен для управленческой картины, но требует ещё одного
  этапа нормализации правил, если на него опираться как на основной financial
  truth.

9. БЕЗОПАСНОСТЬ И ЭКСПЛУАТАЦИЯ
--------------------------------------------------------------------------------

Сильные стороны:
  - есть разделение PUBLIC / TOKEN / ADMIN / PDF
  - есть docs/ACCESS_MODEL.md
  - есть require_admin_session в критичных сценариях
  - есть health/status/check tooling

Риски:
  - разрозненные endpoint'ы требуют постоянной дисциплины
  - токен-ссылки надо считать чувствительными
  - debug HTML и служебные логи не должны быть общедоступны
  - destructive endpoints требуют особенно строгого контроля

Минимальные эксплуатационные правила:
  - регулярный backup storage/
  - HTTPS only
  - контроль прав на storage и config.local.php
  - периодическая проверка health.php

10. ГЛАВНЫЙ БИЗНЕС-ПОТЕНЦИАЛ ПРОЕКТА
--------------------------------------------------------------------------------

Проект уже сейчас может выступать как:
  - order center
  - lightweight CRM
  - service desk
  - inventory-aware repair workflow
  - client communication layer

То есть ценность проекта не в отдельной квитанции или отдельной форме заказа, а
в том, что все ежедневные процессы мастера сходятся в одну систему.

--------------------------------------------------------------------------------
11. ЧТО СЕЙЧАС ВАЖНЕЕ ВСЕГО ДЕРЖАТЬ ПОД КОНТРОЛЕМ
--------------------------------------------------------------------------------

Приоритет 1.
  Стабильность заказного сценария:
  заказ -> OWL -> "Под заказ" -> приход -> AUTO OUT -> Track/status sync

Приоритет 2.
  Нормализация финансовых правил:
  что считать доходом, расходом, прибылью и когда именно

Приоритет 3.
  Консистентность client portal и document links:
  order_id, client_token, viewer links, portal-first policy

Приоритет 4.
  Контроль legacy/fallback логики:
  не допускать возврата fuzzy matching и неочевидных путей сохранения

--------------------------------------------------------------------------------
12. РЕКОМЕНДУЕМЫЙ ПЛАН ДАЛЬШЕ
--------------------------------------------------------------------------------

1. Заморозить и периодически перепроверять критический складской сценарий.
   Это главный operational backbone проекта.

2. Довести finance до формально согласованной модели.
   Отдельно описать:
   - cash metrics
   - order-period metrics
   - estimate metrics

3. Сделать единый короткий "source of truth map".
   Нужен компактный документ: где primary data, где overlay, где fallback.

4. Провести endpoint security pass.
   Просто пройтись по критичным API и проверить:
   admin-only / token-only / public.

5. Ограничить и контролировать debug artefacts.
   Особенно для generated_pdfs/debug_html и чувствительных логов.

--------------------------------------------------------------------------------
13. ИТОГОВАЯ ОЦЕНКА
--------------------------------------------------------------------------------

FixariVan — это уже не сырой прототип, а рабочий монолитный сервисный продукт с
хорошим прикладным покрытием реальных задач мастерской.

Его сильная сторона:
  практичность, цельность и реальный business workflow.

Его слабая сторона:
  сложность связей между заказом, складом, документами и аналитикой.

Если удержать дисциплину в 3 местах:
  - order/inventory consistency
  - finance rules consistency
  - access/security consistency

то проект можно считать очень сильной базой для дальнейшего роста без
обязательного перехода на тяжёлую ERP/CRM архитектуру.
